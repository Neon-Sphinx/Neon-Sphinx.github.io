Skill 撰写方法论：如何写出优秀的 Claude 技能
#

本文档基于对 awesome-claude-skills 仓库中 40+ 个技能的分析总结，旨在帮助初学者快速掌握 Claude Skill 的撰写技巧。

一、什么是 Skill？为什么要写 Skill？
#

简单来说，Skill 就像是给 Claude 装上的"插件"，让它学会特定的专长。

想象一下：

  • • 普通 Claude 像是一个通才，什么都知道一点

  • • Skill 则像是给 Claude 配备的"专业工具包"，让它在特定领域变得专业

写 Skill 的好处：

  1. 1. 复用性强 - 写一次，可以反复使用，甚至分享给团队

  2. 2. 上下文高效 - Skill 采用渐进式加载，只读取需要的内容

  3. 3. 可维护 - 集中管理专业知识，便于迭代更新

三、撰写优秀 Skill 的五大原则
#

原则 1：描述要精准（第三人称原则）
#

YAML 描述字段必须用第三人称：

❌ 错误写法：

<span></span><code id="code-lang-makefile">description:&nbsp;使用这个技能来翻译文章...</code>

✅ 正确写法：

<span></span><code id="code-lang-makefile">description:&nbsp;翻译英文文章为简体中文&nbsp;Markdown&nbsp;格式...</code>

为什么？ 因为描述会被系统读取，用第三人称让系统更准确理解技能用途。

原则 2：指令要用祈使句（动词优先）
#

在 SKILL.md 主体中，用"动词开头"的指令：

❌ 错误写法：

<span></span><code>你应该首先读取文件，然后分析内容...</code>

✅ 正确写法：

<span></span><code>首先读取文件，然后分析内容...</code>

更简洁的版本：

<span></span><code id="code-lang-markdown">1.&nbsp;读取文件2.&nbsp;分析内容</code>

为什么？ 祈使句更直接、更易执行，减少歧义。

原则 3：控制字数（黄金法则：5000 字以内）
#

SKILL.md 推荐字数分级：

技能类型 推荐字数 示例
超简洁型 < 250 字 explaining-code (96 字)
简洁型 250-1000 字 internal-comms (211 字)
标准型 1000-2000 字 article-translator (1053 字)
详细型 2000-5000 字 skill-creator (1660 字)

超过 5000 字怎么办？

  • • 把详细内容移到 references/ 目录

  • • 把代码移到 scripts/ 目录

  • • SKILL.md 只保留核心流程

原则 4：渐进式信息组织
#

信息密度递减的组织方式：

  1. 1. 顶部（必读） - 使用场景、触发条件

  2. 2. 中部（流程） - 具体工作步骤

  3. 3. 底部（参考） - API 文档、详细说明

示例结构：

<span></span><code id="code-lang-perl">---name: my-skilldescription: ...---# 技能标题## 什么时候使用 ← 用户最先看到-&nbsp;需求场景 A-&nbsp;需求场景 B## 这个技能能做什么 ← 快速了解价值1.&nbsp;功能一2.&nbsp;功能二## 使用步骤 ← 核心内容1.&nbsp;步骤一2.&nbsp;步骤二## 示例 ← 实际演示完整的工作流程示例...## 技术细节 ← 按需阅读API 参数、配置等...</code>

#

好的示例应该：

  • • 展示完整工作流程

  • • 包含输入和输出

  • • 覆盖常见场景

#

#

#

#

用第三人称，完整句子：

<span></span><code id="code-lang-bash"># ✅ 好的描述description:&nbsp;翻译英文文章为简体中文&nbsp;Markdown&nbsp;格式。当用户提供英文文章&nbsp;URL&nbsp;并请求翻译时使用此技能。支持保留原有结构和图片。# ❌ 不好的描述description:&nbsp;翻译文章</code>

描述三要素：

  1. 1. 功能 - 做什么

  2. 2. 触发条件 - 什么时候用

  3. 3. 关键特性 - 支持什么

七、文件组织策略
#

7.1 什么时候创建 scripts/？
#

需要脚本的情况：

  • • 代码会被反复调用

  • • 需要确定性执行结果

  • • 逻辑复杂，不适合在文档中重复描述

示例：

<span></span><code id="code-lang-shell"># scripts/my_helper.pydef&nbsp;process_data(input_data):&nbsp; &nbsp; """复杂的数据处理逻辑"""&nbsp; &nbsp; # ...&nbsp; &nbsp; return&nbsp;result</code>

7.2 什么时候创建 references/？
#

需要参考资料的情况：

  • • API 文档（参数、端点等）

  • • 长篇技术规范

  • • 需要按需加载的内容

示例：

<span></span><code data-language-pending="" data-raw-code="references/
├── api_docs.md      # API 完整文档
├── schema.md        # 数据结构说明
└── troubleshooting.md  # 故障排查指南" data-show-line-number="false" id="code-lang-bash">references/├── api_docs.md &nbsp; &nbsp; &nbsp;# API 完整文档├── schema.md &nbsp; &nbsp; &nbsp; &nbsp;# 数据结构说明└── troubleshooting.md &nbsp;# 故障排查指南</code>

7.3 什么时候创建 assets/？
#

需要资源文件的情况：

  • • 输出模板（HTML、Markdown 等）

  • • 图片、字体、样式表

  • • 配置文件模板

  4. 4. 收集反馈： 让其他人试用并收集意见

九、常见问题 FAQ
#

Q1: SKILL.md 太长怎么办？

A: 将详细内容移到 references/ 目录，SKILL.md 中添加引用：

<span></span><code id="code-lang-bash">完整的 API 文档请参考&nbsp;`references/api_docs.md`</code>

Q2: 代码应该放在 SKILL.md 还是 scripts/？

A: 判断标准：

  • • 只展示一次 → 放 SKILL.md

  • • 会反复调用 → 放 scripts/

  • • 复杂逻辑 → 放 scripts/

Q3: 如何让 Skill 更容易被触发？

A: 在描述和使用场景中包含：

  • • 用户可能用的关键词

  • • 同义表达

  • • 常见请求模式

<span></span><code id="code-lang-vbnet">description:&nbsp;翻译英文文章。当用户请求"翻译"、"translate"、"转换为中文"时使用。</code>

Q4: 可以在一个 Skill 中调用另一个 Skill 吗？

A: 技术上可以，但建议保持 Skill 独立性。如果需要共享代码，考虑：

  • • 创建共享的 scripts/

  • • 在 references/ 中引用

Q5: Skill 与 Claude Prompt 的区别？

A:

特性 Skill Prompt
复用性 高（可反复调用） 低（一次性）
上下文管理 渐进式加载 全量加载
版本控制 支持 困难
分享 易于分发 需要复制粘贴

  4. 4. 指令明确 - 祈使句，无歧义

  5. 5. 示例完整 - 覆盖主要场景

  6. 6. 长度适中 - SKILL.md < 5000 字

  7. 7. 易于维护 - 代码与文档分离

  8. 8. 可复用 - 写一次，多次用
